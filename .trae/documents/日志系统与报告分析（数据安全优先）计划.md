## 总目标
- 后端日志：替换 print，建立可观测、可追踪、可检索的专业日志体系。
- 报告分析：Apifox 负责执行；本项目聚焦“报告入库 + 派生指标 + 可视化分析/对比”。
- 数据安全优先：任何频繁迭代都不允许造成数据丢失/污染。

## 关键数据安全原则（贯穿全程）
- **Raw 报告不可变**：MongoDB 中的原始报告文档一旦写入，不做就地覆盖/裁剪；只允许追加字段或写入新集合。
- **派生数据可重建**：趋势/失败明细/性能指标等全部视为“可从 Raw 重算”的派生层；出现 bug 时直接重跑 backfill，而不是改 Raw。
- **写入幂等**：所有写 MongoDB 的派生写入使用 upsert（基于 execution_id + report_id + 唯一键），重复执行不会产生重复/覆盖错误。
- **上线前验证**：每次变更包含“抽取一致性校验脚本”（样本比对：Raw ↔ 派生 ↔ MySQL 摘要字段）。
- **软升级**：新增字段/集合/索引优先；避免破坏性迁移。若必须迁移，采用“双写+回填+切读+保留旧数据”的四步。

## 第一阶段：日志系统完善（后端）
### 1) 统一日志入口
- 新增 `app/core/logging.py`：用 `logging.config.dictConfig` 统一配置。
- 支持环境变量：`LOG_LEVEL`、`LOG_FORMAT=console|json`、`LOG_FILE`（可选）、`LOG_ROTATION`（可选）。
- 统一 uvicorn/fastapi/app logger（uvicorn.error/uvicorn.access/app.*），避免多处 `basicConfig`。

### 2) 请求链路追踪（request_id）
- 新增中间件：生成/透传 `X-Request-ID`，响应头回传。
- 用 `contextvars` + `logging.Filter` 自动把 request_id 注入每条日志。
- 关键字段：method/path/status/duration_ms/client_ip/user_id（登录后）

### 3) 替换 print & 规范化业务日志
- 替换生产代码中的 `print()`（优先：MongoDB 连接、Apifox 执行链路等）。
- 关键业务事件统一事件名（结构化字段）：
  - `apifox_run_started/finished/failed`
  - `mongodb_report_save_ok/fallback_to_mysql`
  - `report_extract_started/finished/failed`
- 严格脱敏：token/JWT/敏感 env 不进日志。

### 4) 验收
- 本地跑通：登录、拉执行列表、获取报告。
- 检查：无 print 输出；每个请求有 request_id；错误可定位到 execution_id。

## 第二阶段：报告分析数据模型（Raw + Derived）
### 1) 保持现有 Raw（不动/不删）
- 继续使用现有 `test_reports`（raw report + metrics + failed_cases）。
- 只允许新增字段（如 schema_version），不删除/不覆盖 report。

### 2) 新增派生层（可重建）
- 新集合（建议）：
  - `report_summaries`：execution_id、collection_id、env、started_at、completed_at、tests_total/passed/failed、rt_avg/min/max、version/schema_version。
  - `report_failures`：一条 failure 一文档（execution_id、api_path、method、status_code、error_message、case_name、created_at）。
  - `report_requests`（可选）：一条请求执行一文档（execution_id、api_path、method、status、latency、assertions…）。
- 这些集合都能从 Raw 重算，允许清空重建。

### 3) 索引与唯一约束（防重复/提性能）
- Raw：execution_id、apifox_collection_id、created_at。
- Summary：唯一键 `(execution_id)`。
- Failures：唯一键 `(execution_id, api_path, method, case_name, failure_fingerprint)`。

## 第三阶段：抽取/回填机制（保证不影响线上写入）
### 1) 执行完成后的抽取（在线）
- 执行完成写 Raw 后：异步触发抽取（后台任务/队列化；无队列则先用轻量后台任务）。
- 抽取写派生集合采用 upsert。

### 2) Backfill（离线可重复跑）
- 提供命令/脚本：按时间范围或 execution_id 列表重算派生数据。
- 每次 backfill 先写入临时集合或带 `run_id` 标记，校验通过后再切换读路径。

### 3) 一致性校验
- 抽样校验：
  - Raw.stats.tests.total 与 summary.tests_total 一致
  - Raw.timings 与 summary 响应时间字段一致
  - failures 数量与 failures 集合一致

## 第四阶段：分析 API（后端）
- 在现有 executions analytics 路由上补齐/统一：
  - 趋势：通过率/失败数/响应时间
  - 高频失败 API（Top N）
  - 性能统计（avg/min/max，后续扩展 p95/p99）
- 新增对比能力（核心价值）：
  - `compare(left_execution_id, right_execution_id)`：新增失败/修复失败/性能回归 TopN
- 所有分析优先读派生集合；派生缺失时可回退从 Raw 计算（只用于小范围/单次）。

## 第五阶段：前端可视化规划
- 新增“报告分析”模块页面：概览/趋势/失败/性能/对比。
- 过滤器：时间范围、集合ID、执行类型、环境、状态。
- 图表：折线（趋势）、柱状（TopN）、表格（明细）、Diff 面板（对比）。

## 变更交付顺序（保证风险最小）
1) 日志系统（不改数据结构）
2) 派生集合与索引（只新增，不改 Raw）
3) 抽取与 backfill（可重复跑，带校验）
4) 分析 API（先读派生，必要时回退 Raw）
5) 前端可视化

## 你关心的“数据不会出现问题”的具体保障
- 不删除 MongoDB 的 raw 报告；派生层即使出错也可整库重建。
- 每一步上线都提供：回滚策略（关闭抽取/关闭派生读，回退到旧接口逻辑）。
- 每次结构变化只做“新增集合/新增字段/新增索引”，避免破坏性迁移。
