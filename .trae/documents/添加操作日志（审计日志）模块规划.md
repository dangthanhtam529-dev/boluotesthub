## 目标与范围
- 目标：把“谁在什么时间对什么资源做了什么操作，结果如何（成功/失败），关联请求与上下文”落库并可查询。
- 范围：先覆盖关键管理/写操作（创建/更新/删除、执行测试、修改配置/用户），再逐步扩展到更多业务动作。

## 数据模型（后端落库）
- 新增表：OperationLog / AuditLog（SQLModel + Alembic 迁移）。
- 核心字段（建议）：
  - 基本：id、created_at、request_id（复用现有请求上下文）、actor_user_id、actor_email（可选冗余）、actor_ip、user_agent。
  - 行为：action（enum：create/update/delete/run/login/…）、resource_type（如 execution/user/report）、resource_id、resource_name（可选）。
  - 变更：before（JSON，可选裁剪）、after（JSON，可选裁剪）、diff_summary（字符串，可选）。
  - 结果：status（success/failure）、error_code、error_message（截断）、duration_ms（可选）。
  - 多租户/环境（如需要）：project_name / apifox_collection_id / environment。
- 索引：created_at、actor_user_id、resource_type+resource_id、action、status（满足按时间/人/资源/动作检索）。
- 数据安全：对敏感字段（token/密码/密钥）做过滤；对 before/after 做白名单字段或长度上限。

## 采集方式（记录点设计）
- 方案 A（推荐，显式打点）：提供 audit service（例如 `audit.log(...)`）供业务路由在关键动作处调用。
  - 优点：语义最准确、可记录 before/after、可控。
  - 落地点：用户管理、执行创建/执行运行、重要配置修改等。
- 方案 B（补充，自动化中间件）：在请求中间件/依赖层自动写“通用请求审计”（method/path/status/duration），并允许路由通过 header/ctx 附加 action/resource 信息。
  - 优点：覆盖面广；缺点：资源语义与 before/after 难准确。
- 结合现状：复用现有 RequestContextMiddleware 的 request_id 与 deps 中的 user_id 注入链路，避免重复造轮子。

## API 设计（后端）
- 新增路由：`/api/v1/audit-logs`（仅 superuser 或具备权限的角色可见）。
- 支持功能：
  - 列表分页：skip/limit。
  - 筛选：时间范围、actor_user_id、action、resource_type、resource_id、status、关键词（resource_name/error_message）。
  - 详情：查看单条日志（包含 before/after/diff）。
  - 导出（可选）：CSV/JSON（限制最大范围、防止大查询）。

## 前端管理页面
- 在侧边栏为 superuser 增加“操作日志/审计日志”入口（沿用现有 /admin 的守卫模式）。
- 页面能力：
  - 列表表格 + 筛选（时间、用户、动作、资源类型、结果）。
  - 行展开/详情抽屉：展示 resource、error、diff/before/after（做折叠与截断）。
  - 一键复制 request_id（用于后端日志关联排查）。

## 日志关联与可观测性
- 每条审计日志保留 request_id，与现有结构化日志字段一致；支持根据 request_id 串联“请求日志 + 审计日志”。

## 权限与合规
- 默认仅 superuser 可读；后续如要开放给项目管理员，补充 role/permission。
- 过滤/脱敏策略：
  - 在审计 service 中统一做字段过滤（password/token/authorization/header 等）。
  - 对 error_message/before/after 做长度上限与字段白名单。

## 数据生命周期
- 提供配置：保留天数（如 30/90/180），并提供后台定时清理脚本/任务（可先做脚本手动跑）。

## 测试与验收
- 单元测试：审计 service 的脱敏/截断、diff 生成（如做）。
- 集成测试：触发关键路由动作后，能在 audit-logs 列表中查到对应记录。
- UI 验收：筛选可用、详情可展开、request_id 可复制、导出受限。

## 逐步落地顺序（建议）
- Phase 1：建表 + audit service + 列表/详情 API + 前端基础列表页。
- Phase 2：覆盖关键写操作（用户/执行/报告相关）并补充 before/after。
- Phase 3：导出、保留策略、更多筛选维度与性能优化（索引/归档）。
