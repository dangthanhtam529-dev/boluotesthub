# Issue Tracker（问题库沉淀）

## 目的

把项目中遇到的各类报错、异常现象、根因与解决方案沉淀为“问题库”，便于你不定期回顾与检索。

## 何时调用（触发条件）

- 出现报错/异常：终端报错、接口报错、前端 NotFound/500、构建失败、数据库异常等。
- 用户提出异常：用户描述“报错/不对/没数据/空白/卡死/权限/登录失效”等。
- 问题已定位/修复：完成修复后立即记录（包含如何复现与如何验证）。

## 文件约定

- 目录：`docs/issues/`
- 文件名：`issues.md`（单文件持续累积，便于全局搜索）
- 追加写入：每次新增一条问题记录，按时间倒序或在文件末尾追加均可；推荐追加到文件末尾并使用清晰标题。
- 禁止写入：任何密钥、token、密码、个人隐私信息。

## 写入格式（模板）

每次新增一条问题，使用如下结构：

```md
## [YYYY-MM-DD HH:MM] {简短标题}

### 现象
- ...

### 影响范围
- ...

### 复现步骤
- ...

### 根因
- ...

### 解决方案
- ...

### 验证方式
- ...

### 相关位置
- backend: ...
- frontend: ...
```

## 执行步骤（给助手的操作指引）

1. 确保 `docs/issues/` 目录存在，不存在则创建。
2. 读取 `docs/issues/issues.md` 末尾（若文件不存在则创建）。
3. 基于本轮对话/终端输出提取一个“问题条目”：
   - 标题要可检索（包含关键错误码/关键词）
   - 复现步骤与验证方式必须最小化、可执行
4. 将条目追加到 `docs/issues/issues.md` 末尾。
5. 在回复中给用户一段简短摘要：记录了什么、关键信息是什么。

## 示例

```md
## [2026-02-13 10:20] 报告分析接口 404 导致页面无数据

### 现象
- 前端 /reports 刷新提示 404 Not Found

### 影响范围
- 报告分析页面无法加载集合列表

### 复现步骤
- 打开 /reports 点击刷新

### 根因
- 前端请求了后端不存在的 /analytics/collections 接口

### 解决方案
- 前端对该接口失败做降级，并改为从执行列表聚合集合ID

### 验证方式
- npm run build 通过
- 页面刷新不再报错

### 相关位置
- frontend: reports.tsx
```
