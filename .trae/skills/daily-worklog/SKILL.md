# Daily Worklog（每日任务整理）

## 目的

把当天完成的任务、关键发现/提示、遗留问题整理成可复用的工作日志文件，避免第二天接续时丢失上下文。

## 何时调用（触发条件）

- **收尾/阶段完成时**：用户说“收尾/总结/今天先到这/把进度记一下/保存一下今天做了什么/下班”等。
- **开始新任务前**：用户说“继续/开始今天/开工/新任务/重启项目后继续”等。

## 文件约定

- 目录：`docs/worklog/`
- 文件名：`YYYY-MM-DD.md`（例如 `2026-02-13.md`）
- 追加写入：同一天多次整理时，在文件尾部追加一个新条目（保留历史）。
- 禁止写入：任何密钥、token、密码、个人隐私信息。

## 写入格式（模板）

每次整理追加如下结构（建议控制在 30~120 行内）：

```md
## {HH:MM} 记录

### 今日完成
- ...

### 关键提示/结论
- ...

### 变更点（文件/模块）
- ...

### 未解决/风险
- ...

### 明日/下一步
- ...

### 复现/验证方式
- ...
```

## 执行步骤（给助手的操作指引）

### A. 开始新任务前（读）

1. 若 `docs/worklog/` 不存在则创建（不报错）。
2. 查找最近 7 天内最新的 `docs/worklog/YYYY-MM-DD.md`。
3. 阅读最新文件，提炼：
   - 上次做到哪里
   - 未解决/风险
   - 下一步入口（命令/页面/接口）
4. 将提炼结果用 5~10 行发给用户，作为“今日开工上下文”。

### B. 收尾/阶段完成时（写）

1. 生成当天日期文件 `docs/worklog/YYYY-MM-DD.md`（不存在则创建）。
2. 以模板追加一条“{HH:MM} 记录”，内容来自：
   - 本轮对话中明确完成的事项
   - 已验证的结果（构建通过/接口返回/页面可访问）
   - 关键限制与注意事项（例如数据源缺失导致无法统计）
3. “变更点”必须列出关键文件 basenames（不需要行号）。
4. “复现/验证方式”用可执行的最小步骤描述（不写密码/密钥）。

## 示例

```md
## 18:40 记录

### 今日完成
- 报告分析页新增执行明细展示（状态码/返回内容/错误提示）

### 关键提示/结论
- 历史报告缺少 executions 明细，状态码从 collection.responseDefinition 推导

### 变更点（文件/模块）
- backend: mongodb_report.py
- frontend: reports.tsx, executions.tsx

### 未解决/风险
- 若要按“真实返回码”统计，需要 Apifox 报告包含 executions 明细

### 明日/下一步
- 调研 Apifox CLI 是否可输出 executions 明细并落库

### 复现/验证方式
- 打开 /reports 查看概览状态码分布
- 在 /executions 复制执行ID并在 /reports 的执行明细页加载
```
